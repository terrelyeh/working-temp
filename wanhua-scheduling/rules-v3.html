01營業時間

萬華世界以週一為全體公休日，其餘六天皆有營業。非週一的國定假日照常開店， 補假日（連假補休）同樣不算國假、不需要特別處理。若國假落在週一，全店仍然公休、且多放一天假（Winnie 4/28 確認，例：除夕落週一）。 平日、長班日、週日的營業時段略有不同，晚班僅在週五六運作到次日凌晨。

日別週幾營業時段備註
公休週一—全體公休；週一遇國假仍公休、多放一天（v4.2 修正自原 v3「不補假」）
平日週二、三、四09:00–19:00
長班日週五、六09:00–次日 00:30含晚班
週日週日09:00–19:00
非週一國假—照常營業套用當日對應日別規則
例外營業（拍攝包場 / 大型活動等）可由主管在月行事曆寫 day_overrides[date].open: true 覆寫週一公休（例：5/11 拍攝包場 Winnie 自上 12h）。

03員工分類

員工依角色分成四類：高階主管（以管理為主、人力不足才下去排班）、 正職（店內主力，月目標工時依公式計算）、PT（每月 56–100h，靠意願表）、 外援（填補剩餘，達 32h 有「喝酒金」福利）。

PT 與外援的給班優先序：一般日子盡量讓所有人工時平均； 但國定假日因為是雙倍薪資、大家都想排，僧多粥少時會以當月意願表送出先後為排序依據——越早送出的越有機會排到國假。

類別月工時排班角色給班優先序成員
高階主管≤ 40h 軟性人力不足才下去排—Winnie / Fish / 政道
正職每月變動（見 §04）主力—超威 / 思坦 / 蛋頭
PT56–100h依意願表國假優先，平日大致平均Jessie / 愛因 / 佳佳 / 91 / Sin / 沛慈 / 小黃
外援達 32h 有喝酒金填補剩餘國假優先，平日大致平均莎賓 / 景文 / 凱柏 / 子寧 / PYY（教學班）

3.1 新人 / 教學班

新進外援的前幾班屬於「教學班」——讓新人試著上班幾次，確認他可以勝任、且願意未來再來。 教學班沒有特殊規則，只是個備註。新人通常還不會收到排班的調查表，所以教學班一定是主管手動安排。

系統處理方式：
  • 員工檔加 note: "新人 / 教學班" 欄位（如 PYY 5/6 起）
  • 月行事曆 pre_assignments 區段由主管直接指派班次（員工、日期、班別）
  • solver 會跳過「沒交 form 整月不可排」的限制並強制排上 pre_assignments 列出的班次

04正職工時計算

正職是月薪制，每月的目標工時不是固定值，而是依當月總天數、每人固定週休、特殊協議、國假與會議計算。 公式把這幾項先扣除得出「月工時總額」，再把當月實際使用的特休與會議時數扣掉，就是真正要 solver 排進班表的「排班目標工時」。

公式

月工時總額 = (當月總天數 − 固定週休天數 − 特殊協議天數 − 非週一國定假日天數) × 8h 排班目標工時 = 月工時總額 − 特休時數 − 會議時數 會議時數 = 2h（每月第一個週二店務例會，固定每位正職）

2026 年 5 月實算範例

正職固定週休上班天數月工時總額特休會議排班目標
超威週一、二 (+1 賣漢堡)21168h16h2h150h
思坦週一、三、四18144h02h142h
蛋頭週日、一、二17136h02h134h
排班目標工時是 solver 實際要排進班表的時數；月工時總額則是員工當月對公司的總時數貢獻（含特休與會議的基準）。

05國定假日補償

非週一國定假日店裡照常營業，所以仍需要有人排班。正職與 PT/外援的補償機制不同： 正職不論是否上班都會獲得 1 天補休，如果真的被排到上班，再加發 double pay； PT 與外援只有真的被排到上班才能領 double pay。

類別不論是否上班若被排上班（加碼）總效益
正職✅ 多補 1 天假（無條件）✅ 再加 double pay≈ 3× 薪資價值
PT—✅ double pay2× 時薪
外援—✅ double pay2× 時薪
正職 form 裡沒勾國假 = 可被排班，不代表一定會上；最終以排班結果為準。 無論上不上班，補休一天是固定有的。

6.3 國假 staffing

套用當天 weekday baseline + 至少 1 位正職。國假落週一 → 全店公休。

6.4 day_overrides 機制

當月特殊狀況（半天營業、活動加人、拍攝包場…）由月行事曆的 day_overrides[YYYY-MM-DD] 區段處理。Override 採完全取代策略——若有 override，當日 staffing 整個用 override 寫的，不混搭 baseline：

  • open: true / false 覆寫週一公休或非週一停業
  • closing_time 覆寫當日打烊時間（5/23 半天案例）
  • staffing 覆寫該日 M8/A8/N75 人數
  • custom_shifts 完全取代 staffing，列出特定員工 + 起訖時間（5/11 Winnie 12h@10:00）

07必須遵守規則 必須遵守

這 8 條是硬規則，排班演算法必須全部滿足才能輸出班表。違反任何一條都視為非法排班， solver 會回報「無可行解、建議放寬哪一條」。其中 H1（兩班 11h 間隔）依據勞基法， 其餘是店內規範。

  1. H1 · 兩班間休息 ≥ 11 小時（勞基法）
    晚班 00:30 下班 → 下次最早 11:30 才能上班（連午班都不行）
  2. H2 · 連續工作天數
    • 任何混排 ≤ 6 天（週一公休自然限制）
    • 純 8h 班 ≤ 5 天
  3. H3 · 同日只排一個班（禁止 split shift）
  4. H4 · 關鍵時段正職下限（見 §06）
  5. H5 · 週一全體不排班
  6. H6 · 主管不取代可排正職（只要還有正職可排，主管不佔用該班）
  7. H7 · PT 月工時 56–100h
  8. H8 · 外援達 32h 有喝酒金（達成門檻，非上限）

08盡量達成目標 盡量達成

軟規則是 solver 的最佳化目標，違反會產生代價分數。演算法會找「總代價最低」的可行排班。 S1 正職達標與 S7 正職同日休假限制權重最高（接近硬規則）， 晚班/長班日公平性、給班順序、個別偏好（如佳佳住遠偏早午班）權重較低但仍要盡量滿足。

09個人偏好處理 偏好

員工在 form 備註欄常寫「5/22 只能上早班」「日只能早」這類時段偏好。 系統會根據關鍵字自動解析：「只能」視為硬性（其他班次排除）， 「希望」或「偏好」視為軟性（盡量滿足）。時間限制如「18:00 後」會當作硬性最早開始時間。

關鍵字處理方式範例
只能 X 班硬性（排除其他班）佳佳「只能早班或中班」
希望 / 偏好 X 班軟性（盡量滿足）思坦「早晚班輪流」
X 時後硬性最早開始時間子寧「5/9 18:00 後」
日只能 X該週幾所有日期硬性沛慈「日只能早」

10休假管理

休假資料是工時計算的主要變動來源，分為年假（特休）與特殊假兩大類。 年假採「累積制」——未休完的年度額度會併入次年可用，但只延展 1 年，再未休即作廢。 特殊假按勞基法，天數、給薪、以及是否計入月工時達成列於下表。

10.1 年假（特休）

  • 累積制：未休年假自動累計至次年可用
  • 額度計算範例：
    第 1 年：+3 天 = 可用 3 第 2 年：+7 天，若第 1 年 3 天未用 → 可用 10 第 3 年：+10 天，若累計未用 → 可用 20
  • 無月使用上限——員工每月可請任意天數
  • 半天單位：4 小時
  • 跨月請假：需提前 1 個月通知
  • 到期處理：只延展一年，延展後仍未用即作廢
  • 臨時使用：通常是其他正職臨時請假、時數不足時協議用特休補

10.2 特殊假（按勞基法）

假別天數上限給薪計入正職工時達成

喪假（父母、配偶）8 天全薪✅
喪假（祖父母、子女、配偶父母）6 天全薪✅
喪假（曾祖、兄弟姊妹、配偶祖父母）3 天全薪✅

12跨月補時數

正職月工時少於目標時，下月必須補回不足時數——不論差多少都要補。 但補的過程仍要維持每週休假 1–2 天，不會為了補工時讓員工全勤。 若超過目標，會發加班費，不會在下月少排。Winnie 與 Fish 負責判斷與計算補時數。

  • 少於目標：不管少多少都要補
  • 補時數上限：仍維持每週休 1–2 天
  • 超過目標：發加班費，不會下月少排
  • 計算者：Winnie + Fish 協商
範例：2026/1 蛋頭少 49h（請假陪先生回加拿大），先用年假抵扣，不足時數於 2 月多排補回。

13缺班兜底順序

排完班仍有缺口、或發布後有人臨時請假時，依順序尋找替補： 先問群組大家有沒有人能加班，若無則 Winnie / Fish 下去排，最後問外援凱柏。 這個順序反映「優先動用已排員工、最後才動用較不固定的人力」的原則。

  1. 群組問大家能否加班
  2. Winnie / Fish 下去排
  3. 最後問凱柏（外援）

14活動與班表動態調整

店裡常有特殊活動，有些活動需要加人、有些只要指定某位員工必到、 有些只是提醒。為了讓排班系統能正確處理，活動分四個層級，每個層級系統的行為不同。 理想狀態是：已知活動寫入當月行事曆，solver 排班時就一次納入； 排完才出現的臨時活動，才進 post-schedule 手動調整（留 log）。

14.1 活動分級

層級定義對排班的影響範例
internal 內部提醒，不影響排班 只在班表上標示，不加人 店務例會
meeting 特定人要參與的會議 指定人員的會議時數 +X 小時，不影響店內人力 財務會議、WIP 會議
event_light 需要全員知情 / 指定人員必到 被指定的員工強制排那天那個班，人數不變 瀧自慢社長來訪（思坦必到）
event_heavy 需要加人力 當天班次人數 +N，可能指定人員 新富有樂市、快閃大廚（蛋頭賣麵）

14.4 累積疲勞防護

活動加班後，solver 在後續日子計算排班時，會考慮該員工「最近累積時數較多」， 降低在往後幾天被重複排晚班或長班日的機率。避免因應活動而讓個別員工過度疲勞。

15換班流程

班表公告後，員工想換班需經過正式流程：發起人 → 接手人同意 → Winnie 核准 → 系統更新。 這確保換班紀錄可追溯。通知對象則分兩類：店內臨時改動（如包場）通知全員； 員工個人私下調班只通知受影響者，避免訊息雜訊。

  1. 員工 A 發起換班、指名員工 B
  2. 員工 B 同意
  3. Winnie 核准
  4. 系統更新班表並留異動紀錄

通知規則

  • 店內臨時改動（包場、活動）→ 通知全員
  • 同事個人調班 → 只通知受影響者

16個人特殊協議

部分員工有個人化協議（例如超威「賣漢堡月多休 1 天」），但這些不是永久設定， 而是逐月確認。每月行事曆裡以 special_agreements 欄位登記「本月是否適用」、 哪幾天作為該協議對應的休假日。這樣才不會每月無差別套用、導致工時計算錯誤。

員工協議適用範圍
超威賣漢堡月 → 額外 +1 天休假視當月是否賣漢堡逐月確認
之後若有其他人的類似協議（例如「X 月進修」、「X 月回老家」），都透過月行事曆的 special_agreements 欄位登記，不寫死在員工檔裡。

18v4 系統機制概覽

v4 引入三個機制處理「規則無法用單一 baseline 涵蓋的個別狀況」。這些機制不改變業務規則，只是把規則的特例有結構地存進月行事曆。

18.1 day_overrides — 單日特殊安排

用於：半天營業、拍攝包場、活動 staffing override、週一例外開店等。寫在 data/YYYY-MM/calendar.yaml。

欄位用途範例
open覆寫該日開 / 關店5/11 拍攝包場 → open: true
closing_time覆寫打烊時間5/23 → closing_time: "18:00"
staffing取代 baseline 人數5/23 → {M8: 2, A8: 3}
custom_shifts非標準班次（含起訖時間 + 指派人）5/11 → Winnie 10:00–22:00 12h

18.2 pre_assignments — 主管硬指派

用於：教學班（新人）、特殊情況的主管直接指派。Solver 不會推翻這些指派。

pre_assignments: - employee: PYY date: 2026-05-06 shift: A8 note: "教學班 day 1"

18.3 post_form_updates — Form 截止後補登

用於：員工晚交 form、form 截止後的個別調整（換班 / 加特休 / 修偏好）。

type作用範例
shift_preference加單日偏好5/2 佳佳 → 只能早班
leave加休假日5/6 超威 → 用特休
late_form_submission整套補交 form 內容莎賓 8 個可排日 + N75 偏好
Phase 2 web app 上線後，post_form_updates 會變成 form 截止後的 audit log；此處先用 yaml 暫存。
